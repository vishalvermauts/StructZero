# IDE Architect MCP Plugin v3.0

The **IDE Architect MCP Plugin** is a fully autonomous, intelligent backend for your IDE, powered by the Model Context Protocol (MCP). It is designed to act as a "Persistent Brain" and a "Principal Architect," helping you design systems, manage persistent project memory, and observe AI telemetry seamlessly within your development environment.

## 🚀 Core Features

### 1. 3-Way Multi-Agent Debate Engine
When you request a new feature or architectural blueprint, the system doesn't just rely on one LLM. Instead, it triggers a multi-agent debate:
- **Round 1 (Gemini 2.5 Pro):** Analyzes your prompt, pulls context from GitHub, and drafts an initial architecture blueprint.
- **Round 2 (Claude 3.5 Sonnet):** Acts as a ruthless Senior Staff Engineer, aggressively critiquing Gemini's draft to find scaling bottlenecks, security flaws, and missing guardrails.
- **Round 3 (DeepSeek):** Synthesizes the original draft and the critique into a perfect, final, production-ready architectural blueprint. (Includes an automatic fallback to Gemini if DeepSeek fails).

### 2. Persistent Memory Bank (Semantic Search)
Instead of starting every coding session with amnesia, the IDE Architect MCP introduces a **Vector Database Memory Bank**. 
- Using Google's `text-embedding-004`, every memory stored is converted into semantic vector embeddings.
- **MCP Tool `store_memory`:** Your IDE agents can permanently save critical decisions, SSH keys, or project state.
- **MCP Tool `search_memory`:** Your IDE agents can semantically query past decisions without needing exact keyword matches.

### 3. AI Observability & Telemetry
A gorgeous, built-in Enterprise Dashboard provides real-time visibility into your AI operations:
- Tracks total LLM API costs down to a fraction of a cent.
- Monitors token usage across different providers (Gemini, Claude, DeepSeek).
- Calculates average latency and provides visual distribution charts using `Recharts`.

### 4. Skills Library
Any architectural blueprint generated by the Debate Engine can be saved as a "Skill." These skills act as templates and reusable context blueprints that the IDE can reference instantly via the `workspace://context` MCP resource.

## 🏗️ Architecture & Tech Stack

### Frontend (React + Vite + Tailwind)
- **UI Design:** Built with React and styled using Tailwind CSS, featuring a stunning dark mode aesthetic (`slate-900`), glassmorphism effects, and neon accents.
- **Routing:** Handled via a dynamic state-based tab layout (Debate Engine, Observability, Memory Bank, Skills, Settings).
- **WebSockets:** Uses `socket.io-client` to display real-time terminal stream logs of the background AI agents working.

### Backend (Node.js + Express)
- **API Server:** An Express server (`server.js`) that handles settings, memory, skills, and orchestrates the complex multi-agent LangChain workflow.
- **Database:** SQLite (`database.js`) acts as the local persistent storage layer for Settings, Memories (with stringified vector arrays), Skills, and Observability Metrics.

### MCP Server (`mcp.js`)
- Exposes tools to any MCP-compliant IDE (like Claude Desktop or Cursor).
- **Resources Exposed:** `workspace://context` (Injects the active architecture and memory bank into the LLM context window).
- **Tools Exposed:**
  - `store_memory(topic, details)`
  - `search_memory(query, limit)`
  - `troubleshoot_error(errorLog)`

## ⚙️ How It Works (The Lifecycle)

1. **Setup:** You enter your API keys in the Settings tab. They are securely saved to the local SQLite database.
2. **Generating a Blueprint:** 
   - You type a prompt into the Debate Engine.
   - The frontend calls `/api/generate`.
   - The backend runs a background semantic search on GitHub to find relevant real-world context.
   - It pulls historical context from the Vector Memory Bank.
   - The 3-way LLM debate executes sequentially. Real-time updates are streamed back to the frontend via WebSockets and displayed in the "Live Agent Stream" ticker.
   - The final architecture is rendered in a Markdown viewer.
3. **Saving to Memory:** You can save the blueprint to the Skills Library, or the IDE agent itself can call `store_memory` to permanently log facts.
4. **Monitoring:** Every LLM call logs its latency, token count, and cost. You can view these metrics visually on the Observability Dashboard.

## 🐞 Troubleshooting

- **Double Scrollbars or Menu Hiding:** The UI is strictly optimized with `overflow-clip` and `scrollbar-hide` utilities to ensure a flawless native-app feel on desktop environments.
- **Missing Memories from Agent Calls:** If an AI agent calls the MCP server, ensure it is configured to point to the exact same directory and SQLite database as the running Express server, otherwise it may write to a sandboxed virtual database instance.
